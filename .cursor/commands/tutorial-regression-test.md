# チュートリアル回帰テスト手順

このドキュメントは、技術チュートリアルの回帰テスト（Regression Test）を行うAIエージェント向けの指示書です。

## 1. 目的

指定されたチュートリアル記事（Markdown）の内容が最新の環境でも正しく動作し、記述に誤りがないかを検証します。

## 2. 実行プロセス

### 2.1. ユーザーシミュレーション

- **読者になりきる**: 知識ゼロの読者がチュートリアルを読み進める状況をシミュレートしてください。
- **順次実行**: 手順を飛ばさず、上から下へ順番に実行してください。
- **段階的編集**: コードが一気に完成形になるのではなく、手順の中で徐々に修正されていく場合、その過程も忠実に再現し、各ステップでの動作確認を行ってください。いきなり最終コードを適用してはいけません。

### 2.2. 検証ポイント

- **記述と実際の乖離**: チュートリアル内のコマンド出力、画面キャプチャ、コードの挙動と、実際に実行した結果が一致することを確認してください。
- **隠された指示 (`regression test`)**: ドキュメント内に `<!-- regression test: ... -->` という形式のHTMLコメントがある場合、それはテスターへの重要指示です。必ずその内容にしたがって確認を行ってください。このコメントは複数行にわたる場合があります。

### 2.3. 実行環境と作業ディレクトリ（重要）

- **実行環境**: 回帰テストは Docker コンテナ内で行います。現在の環境はDockerコンテナ内です。
  - **port**: `3000` が利用できます（必要なら開発サーバー等を起動して確認してください）。
- **作業ディレクトリ**: すべての作業は `/home/developer/regression-test/workspace` 配下で行ってください。
  - チュートリアルの途中でプロジェクトを作る場合も、必ずこのディレクトリ配下に作成してください。
  - 既存ファイルと衝突しないよう、可能なら `workspace/<チュートリアル名>/...` のようにサブディレクトリを切って作業してください。
- **安全性**: コンテナ内の作業なので、多少環境を壊してしまっても問題ありません（作り直せます）。
  - `workspace/` と `issues/` 以外は read-only マウントになっているため、意図しない変更がホスト側に反映されにくい安全な構成です。

## 3. 問題報告

テストの結果、不具合や不整合が見つかった場合のみ報告を作成してください。正常に動作した項目は報告不要です。

- **出力先**: プロジェクトルートの `issues/[チュートリアルファイル名].md`
  - 形式: `project-root/issues/[tutorial filename].md`
  - 例: 対象が `docs/tutorials/nextjs.md` の場合 → `issues/nextjs.md`
  - `issues/` ディレクトリやファイルが存在しない場合は作成してください
- **報告フォーマット**:

```markdown
### チュートリアル: [ファイル名]

#### 不具合箇所: [セクション名] (行番号: [番号])

- **期待される動作**: [チュートリアルの記述]
- **実際の動作**: [検証結果・エラー内容]
- **詳細**: [差異の理由やスクリーンショットとの違いなど]
```

## 4. 対象

- 指定されたファイル、または現在アクティブなMarkdownファイルを対象とします。
